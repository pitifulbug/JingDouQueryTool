# JingDouQueryTool v3.2.2 升级说明

发布日期：2026-07-13

## 一、升级概述

`v3.2.2` 是质量保障与发行工程升级，重点是建立可信、可持续执行的自动化测试与覆盖率门槛。

本次没有修改查询业务逻辑、UI 功能、接口地址、页面匹配规则或扩展权限，`v3.2.1` 的使用方式和业务结果保持兼容。

## 二、变更明细

### 1. 修复覆盖率统计失真

原测试通过独立 VM context 加载生产脚本，Node.js 覆盖率报告无法归因到源码文件，最终出现“没有文件明细但显示 100%”的失真结果。

`v3.2.2` 新增可计量的测试脚本加载器，使 `src/**/*.js` 能进入 Node.js 原生覆盖率报告。

### 2. 扩充自动化测试

自动化测试由 7 个增加到 48 个，新增覆盖以下高风险路径：

- 后台请求代理的域名、路径、请求方法、Header、Body 和重定向安全边界；
- JPOS 账号分组、缓存合并、失败缓存淘汰、并发分页、日期与表格解析；
- kfuad 并发分页、终止页过滤、请求协议及登录和接口错误分类；
- 扩展上下文失效、后台消息通信、超时钳制和错误恢复；
- CRM 组织 ID 识别、多页数据读取及加载失败后的 UI 状态恢复；
- CRM 表格解析、事件号去重和本地导出降级；
- CSV 公式注入防护和筛选结果导出；
- 查询结果 HTML 转义，防止未转义内容进入页面；
- 独立页面模式下表单实时值的保留；
- UI 模板关键元素完整性和窗口状态切换。

### 3. 增加标准测试命令

仓库新增以下零依赖测试入口：

```powershell
npm test
npm run test:coverage
```

如果 PowerShell 执行策略阻止 `npm.ps1`，可以使用：

```powershell
npm.cmd test
npm.cmd run test:coverage
```

### 4. 增加覆盖率回退门槛

| 指标 | 当前结果 | 最低门槛 | 状态 |
| --- | ---: | ---: | --- |
| Lines | 62.42% | 60% | 通过 |
| Branches | 69.97% | 65% | 通过 |
| Functions | 66.67% | 60% | 通过 |

当后续变更导致覆盖率低于门槛时，`npm run test:coverage` 将返回失败，防止覆盖率无提示回退。

## 四、升级步骤

1. 下载并解压 `JingDouQueryTool-v3.2.2.zip` 到一个新目录。
2. 在 Chrome 地址栏打开 `chrome://extensions`。
3. 开启右上角的“开发者模式”。
4. 停用旧版 `v3.2.1`，避免两个版本同时运行。
5. 点击“加载已解压的扩展程序”。
6. 选择新目录中包含 `manifest.json` 的文件夹。
7. 确认扩展卡片显示版本 `3.2.2`。
8. 刷新已经打开的 JPOS 和 CRM 页面。
9. 完成下方的升级后验证。

建议在新版验证完成前保留旧版目录，不要直接覆盖或删除旧版文件。

## 五、升级后验证

建议至少执行以下检查：

1. 京豆查询页面能正常显示工具面板，标题版本为 `v3.2.2`；
2. CRM 数据能够正常读取，人员和日期范围可以选择；
3. JPOS 查询能够启动、停止并显示命中或未命中结果；
4. kfuad 查询能够正常返回结果或明确提示登录和接口错误；
5. CSV 导出文件能够正常打开，筛选结果与页面一致；
6. CRM 本地导出按钮能够正常生成文件；
7. 浏览器扩展管理页没有显示 manifest 或 service worker 错误。

## 六、回滚方法

如果新版在实际环境中出现阻断问题：

1. 在 `chrome://extensions` 停用或移除 `v3.2.2`；
2. 重新启用旧版 `v3.2.1`，或重新加载旧版解压目录；
3. 刷新 JPOS 和 CRM 页面；
4. 保留错误信息、复现步骤和浏览器控制台日志用于定位。

本次不涉及数据迁移，因此回滚不需要执行数据恢复。
